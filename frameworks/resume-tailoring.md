# Resume Tailoring — ATS / Recruiter / HM 三版改写法

三个版本共享**同一份事实**，只是组织、排序、措辞不同。**所有改写必须基于用户简历已有事实，绝不杜撰。**

---

## 零 · 共用原则（三版都遵守）

1. **不杜撰事实**——不能编新成就、新数字、新职责。
2. **不夸大动词**——"参与" 不能改成 "主导"。这条违反招聘经理会在面试里 follow-up 时拆穿。
3. **关键词原样用 JD 的词**——招聘经理 / ATS 都在搜 JD 里的词。用户简历里写 "improved performance"，JD 写 "optimized latency"，改成 "optimized latency"（如果事实匹配）。
4. **数字保留 / 强化**——能加数字的地方加，不能加就保留原有的。
5. **动词排序：影响 > 动作 > 工具**——
   - 弱："Used Figma to design new dashboard"
   - 强："Shipped new dashboard, cutting time-to-insight 40% for 500+ enterprise admins (built in Figma)"
6. **简历是 1-2 页，每条改写要砍掉同等长度的弱内容**——不要无限堆砌。

---

## 一 · ATS 版（机器友好）

### 目标
通过简历筛选系统（Applicant Tracking System）的关键词匹配。

### 必做

- **关键词出现位置和密度**
  - JD 的 Must Have 关键词，在简历里出现 **2-4 次**最理想（少了过不了筛，多了被识别成 keyword stuffing）
  - 关键词必须出现在 Title / Skills / 至少 1 条 bullet 里
  - 同义词都加（如 JD 写 "PM"，简历同时写 "Product Manager" 和 "PM"）
- **结构标准化**
  - Section 标题用标准词：Experience / Work Experience / Education / Skills / Certifications
  - 日期格式统一：MMM YYYY – MMM YYYY 或 YYYY – YYYY
  - 公司名 / Title 单独成行
- **格式避坑**
  - 不用双栏 / 表格 / 图表（很多 ATS 解析不了）
  - 不用 header / footer 放重要信息
  - 不把关键内容做成图片
  - 字体用标准：Arial / Calibri / Times / Helvetica
- **Skills section**
  - 把 JD 列的硬技能词汇都放进去（用户真有的）
  - 工具 / 软件 / 编程语言用 JD 里的写法
- **缩写处理**
  - 既写全称又写缩写：`Search Engine Optimization (SEO)`

### 不做

- 不要为关键词而硬塞——不真实的关键词出现在 bullet 里，HM 一眼看出来
- 不要堆叠同一关键词在多个 bullet 末尾
- 不要做花哨排版

### Checklist
- [ ] JD 的 Top 5 Must Have 关键词都在简历里出现至少 2 次
- [ ] Skills section 涵盖 JD 列的所有硬技能
- [ ] 没有图表 / 双栏 / 复杂格式
- [ ] Section 标题是标准词

## 二 · Recruiter 版（30 秒扫读）

### 目标
让 Recruiter / 内推人在 30 秒内得出"这人匹配，往下传"。

### 必做

- **顶部加 Headline / Summary（1-3 行）**
  - 一句话把用户定位成 JD 在找的那种人
  - 句式：`[Title] with [年限] experience in [JD 的关键领域], having shipped [关键作品 / 量化结果]`
  - 例：`Principal Product Designer with 10+ years shaping AI-powered enterprise products, having led 0→1 design for Microsoft Teams AI (1M+ DAU).`
- **经历排序**
  - 最相关的经历置顶，即使不是时间最新（如果不是时间最新，加副标题"Most Relevant Experience"）
  - 不相关的早期经历压缩到 1 行
- **每段经历的前 2 个 bullet 必须直接对齐 JD 的 Must Have**
  - 第 1 个 bullet = 影响力 / 量化结果
  - 第 2 个 bullet = 端到端 / scope 体现
  - 后面再放其他动作
- **加粗 / 视觉重点**
  - 每段经历可加粗 1-2 个最关键短语（数字 / JD 关键词）
  - 不要全文随便加粗，否则失去强调效果
- **数字前置**
  - "Cut onboarding time by 35% (from 12 to 8 min) across 200K monthly users"
  - 比 "Improved onboarding by optimizing the user flow" 强 10 倍

### 不做

- 不要写求职信式的"objective"句子
- 不要列 10 年前不相关的项目
- 不要密密麻麻没有视觉锚点

### Checklist
- [ ] 顶部 Summary 一句话能让 Recruiter 立刻知道"这人匹配什么"
- [ ] 每段经历前 2 个 bullet 数字 + JD 关键词都到位
- [ ] 不相关经历已压缩
- [ ] 关键短语有视觉强调（加粗 / 数字前置）

## 三 · Hiring Manager 版（体现思考与判断）

### 目标
让招聘经理看出"这人怎么思考、怎么决策、怎么推动事情"。

### 必做

- **Bullet 句式：「面对什么 → 做了什么判断 → 推动什么动作 → 取得什么结果」**
  - 弱：`Designed onboarding flow for new enterprise customers`
  - 强：`Identified onboarding drop-off (60% at step 3) as top churn driver; redesigned to single-screen flow after testing 3 variants, cutting drop-off to 22% and recovering ~$2M ARR`
- **突出 Hidden Signal 对应的特质**
  - JD 强调 ownership → 写"defined direction with"、"drove decision on"、"owned end-to-end from X to Y"
  - JD 强调 ambiguity → 写"in absence of clear requirements, framed problem as"、"established first measurement framework for"
  - JD 强调 influence → 写"aligned 4 teams on"、"convinced leadership to invest in"
- **决策 / 取舍动词**
  - drove, defined, framed, aligned, influenced, shipped, scaled, established, recovered, prevented
  - 比 designed / built / worked on / collaborated 强得多
- **保留 1-2 个"hook 细节"**
  - 招聘经理会在面试里 follow-up 的钩子
  - 例：bullet 里写"cut drop-off to 22%"——面试官会问"为什么是 22%？怎么测的？为什么不是 15%？"——这些 hook 让 HM 觉得"这人有真东西"
- **体现 craft / 反思**（如果是设计 / 工程岗）
  - 加 1-2 个"我是怎么做这个决定的"细节
  - 不要纯流程描述

### 不做

- 不堆 buzzword（synergy / leverage / disrupt）
- 不写虚的（"passionate about UX" / "team player"）
- 不写所有人都能写的（"collaborated with cross-functional teams"——这是默认值）

### Checklist
- [ ] 至少 5 条 bullet 是「问题 → 判断 → 动作 → 结果」结构
- [ ] 关键决策 / 取舍的动词清晰
- [ ] 有 1-2 个能让 HM 想 follow-up 的 hook 细节
- [ ] 没有空洞 buzzword 和"团队合作"这种废话

## 四 · Diff 输出标准

每条改写必须给 diff，让用户学到"为什么这么改"：

```
【职位 X · 改动 #N】

原文：
> [一字不差贴原文]

改后（HM 版 / Recruiter 版 / ATS 版）：
> [新版本]

对齐了 JD 的：
- [Must Have / Hidden Signal / Nice to Have] + 编号
- 关键词："X"、"Y"

依据简历事实：[用户原简历哪句话提供了这个事实，确认没杜撰]
```

**改了几条就给几条 diff。** 用户不需要看完整新简历，需要看到改动和理由。

## 五 · 简历整体结构建议（三版共用顺序）

通用顺序（针对 Senior+ 求职者）：

1. **Name + Contact**（含 LinkedIn / Portfolio）
2. **Headline / Summary**（Recruiter 版 / HM 版必加；ATS 版可选）
3. **Experience**（按相关性 / 时间排序）
4. **Education**（Senior 后可压到 1-2 行）
5. **Skills**（ATS 版重点，HM 版可弱化）
6. **Selected Projects / Publications**（如适用）

---

## 六 · 边界提醒

- **这个 skill 只做 tailor，不做简历整体优化。** 用户想做"简历整体怎么写更好"——告诉他去 Resume Skill。
- **三个版本都建议保留。** 不同投递场景用不同版本。
- **跨语言简历**：中英文简历不能简单互译，文化和招聘场景不一样，要分别 tailor。
