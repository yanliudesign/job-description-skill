# Match Rubric — 匹配度评分规则 + Gap 三档分类

打分必须按这套规则，**不能凭感觉给 92%**。所有评分都要能复现：把简历和 JD 给另一个人按同一规则算，结果应该接近。

---

## 一 · 总分公式

```
Match Score = 0.6 × MustHaveScore + 0.2 × NiceToHaveScore + 0.2 × HiddenSignalFit
```

权重不是绝对的，但要写明用了什么权重。如果某条 must have 是 deal-breaker（如某 cert / 某具体行业），其权重要单独加大或直接做"门槛"——不命中就总分自动 < 50%。

---

## 二 · 单条评分（每条 Must Have / Nice to Have 都套这个三档）

### ✅ 完全命中 = 1.0

满足全部三个：
- 简历有**直接证据**（具体项目 / 经历），不是宽泛 claim
- 有**量化或可验证细节**（数字 / 规模 / 影响）
- 经验**新鲜**（近 3 年内为主，不是 10 年前的）

### 🟡 部分命中 / 相邻经验 = 0.5

满足下列任一：
- 做过相邻领域（如 JD 要 enterprise AI，用户做 consumer AI）
- 简历有 claim 但缺细节 / 缺数字
- 经验偏旧（3-7 年前为主）
- 做过但角色不是 lead（如 JD 要"led X"，用户是"contributed to X"）

### ❌ 未命中 = 0.0

简历**完全看不出**有这块经验。即使用户口头说"我会"，**简历没体现 = 0 分**（招聘经理只看简历）。

---

## 三 · MustHaveScore 计算

```
MustHaveScore = Σ(每条 Must Have 的命中分) / Must Have 总条数
```

例：4 条 Must Have，分别 1.0 / 1.0 / 0.5 / 0.0 → MustHaveScore = 2.5/4 = 0.625

### 特殊规则

- 任何一条 Must Have = 0 → 总分**最高不能超过 75%**（一条 must have miss 已经是大问题）
- 两条以上 Must Have = 0 → 总分最高不能超过 55%
- 出现"门槛型 must have"（如必须有 PR / 必须某 cert）不命中 → 总分**直接打到 25-35%**

---

## 四 · NiceToHaveScore 计算

```
NiceToHaveScore = Σ(每条 Nice to Have 的命中分) / Nice to Have 总条数
```

权重已经在总分公式里减半（×0.2）。

---

## 五 · HiddenSignalFit 计算

这部分主观度高，但要按维度打分，不能整体拍脑袋。从 Hidden Signals 里挑 3-5 个最强的，每个 0 / 0.5 / 1 打分：

- 用户简历 / 画像**明显匹配**该信号 = 1
- 中性 / 看不出来 = 0.5
- 明显不匹配 / 反信号 = 0

例：
- Hidden Signal: ownership / drive
  - 用户简历多个"led / drove / defined" → 1.0
- Hidden Signal: ambiguity / 0-1
  - 用户全是大厂成熟产品经历 → 0.5（不算反信号，但不强）
- Hidden Signal: mission-driven
  - 简历 / 用户陈述里没有 mission-related 内容 → 0.5

```
HiddenSignalFit = Σ / 总数
```

---

## 六 · 输出规则

**永远给区间，不给单点。** 因为：

- Must Have 翻译本身有判断成分（同一条简历事实换个 Decoder 可能打 0.5 也可能打 1）
- Hidden Signal 主观度更高

合理区间宽度：±5-8 个百分点。

```
匹配度 = 72-80%
```

并标 **定性档**：

| 区间 | 档位 | 含义 |
|---|---|---|
| 85%+ | 直接对口型 | 几乎不用 tailor 也能拿面 |
| 70-85% | 强匹配 | tailor + 内推 能显著提升拿面率 |
| 55-70% | 中等匹配 | 有 1-2 个明显 gap，要补 + tailor |
| 40-55% | 弱匹配 | 投不投得看用户机会成本 |
| < 40% | 不匹配 | 不建议投，除非有特殊路径（强内推 / 转岗） |

---

## 七 · Gap 三档分类

每个 Gap（=未命中 / 部分命中的 Must Have 或 Nice to Have）必须标档：

### 🔧 能补（30 天内能弥补）

特征：
- 简历有相邻经验，只是没突出 / 没用对词
- 缺的是 framing / 翻译 / 重新组织
- 缺的是某个 cert / 工具 / 短期能学的硬技能
- 缺的是 portfolio case，能从已有项目里拆一个出来

补法要**具体**："把 X 项目里的 Y 部分单独拆成一个 case，加这 3 条数据"。

### ⏳ 难补（短期补不上）

特征：
- 缺 N 年某行业经验
- 缺 leadership 等需要时间积累的能力
- 缺特定规模 / 特定阶段的经验
- 缺学历 / 国籍 / visa 等结构性条件

补法：要么承认 + 走"故事补齐"路线（用相邻经验 + 学习速度的故事），要么放弃这个 JD。

### 🤷 不重要（JD 写了但实际不卡）

特征：
- 是 nice to have 且不是主线
- JD 例行公事写的（如所有岗位都列的 communication skills）
- 与 must have 重复 / 关联度低

→ 不要在最终 Gap 里列这些，**列了反而干扰用户判断**。

---

## 八 · 风险（区别于 Gap）

Gap = "简历缺什么"。**Risk = "招聘经理看简历会担心什么"**。这是两件事：

例：用户简历是 Microsoft Principal 10 年，投一个 50 人 startup。
- Gap 可能很少（资历足够）
- Risk 很大（HM 担心"大厂能不能适应"、"会不会嫌薪资低"、"会不会觉得 scope 太小"）

Risk 维度清单：

1. **背景跨度风险**：行业切换 / 公司规模切换 / 角色切换
2. **稳定性风险**：跳槽频率 / 短期 stint / gap year
3. **Level 跨度风险**：跨 level 投（升或降）
4. **方向匹配风险**：过往主要 focus 跟 JD 不完全一致
5. **薪资 / 期望风险**：HM 担心用户接受不了 offer
6. **Visa / 地理 / 时区风险**：JD 隐含的硬约束

每个 Risk 必须给：**HM 会怎么质疑** + **用户可以怎么回应**（最好关联到一个故事题材）。
