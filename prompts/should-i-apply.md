# Should I Apply?

**目标：** 输出投不投这份 JD 的决策建议。五件套：推荐指数 / 值得投 / 不值得投 / 拿面概率 / 建议下一步。

**铁律**
- **必须给负面理由**。只说好话的工具没人信。如果实在没有，就老实说"这份没明显的不值得投点"，但要提示用户可能的盲区。
- **拿面概率给区间不给单点**。预测拿面是高方差事件，给单点数字（"70%"）容易被打脸，给区间（"50-70%"）更诚实。
- **不替用户做决定**。给推荐 + 理由 + 建议，但最终用户自己定。

---

## Step 1 · 收齐输入

1. **JD Decode + Match Score 产出**（强烈推荐先有这两个）。没有就先跑解码 + 匹配 的极简版。
2. **用户硬约束**（必问）：
   - 期望薪资底线
   - 地点 / 远程是否能接受
   - 行业 / 公司类型偏好或排除
   - 入职时间窗口
3. **用户当前手里在 pipeline 里的机会数**（可选，影响"机会成本"判断）。

---

## Step 2 · 五件套产出

### ① 推荐指数（⭐ 1–5）

打分依据见 [../frameworks/go-no-go.md](../frameworks/go-no-go.md)。一句话定性：

```
⭐⭐⭐⭐☆（4/5 · 值得投，但建议先 tailor + 找内推）
```

不要轻易给 5 星——5 星 = "几乎没有 downside 的机会"。

### ② 为什么值得投（3 条）

每条要**具体、可验证**，不能是泛泛而谈"公司不错"。挑这些维度里最强的 3 条：

- **方向**：行业 / 技术方向是不是上升期？跟用户长期路径合不合？
- **匹配**：Match Score 高在哪？强项命中 must have 哪几条？
- **公司 / 团队**：背书 / 增长 / 团队画像 / 老板背景（如果有公开信息）
- **薪资 / Level**：相对用户当前是上升 / 持平 / 下降
- **学习曲线**：能学到什么新东西
- **网络价值**：进去能认识什么人 / 拿到什么 brand
- **机会稀缺性**：这类职位多不多 / 多久才有一个

```
✅ 值得投 1：AI Product Designer 方向当前是稀缺岗位，且公司刚拿 C 轮预算充足
✅ 值得投 2：Match Score 78%，强项直接命中 Must Have #1 #3
✅ 值得投 3：HM 之前在 Figma，团队美学和 craft 标准高，对个人成长是加速器
```

### ③ 为什么不值得投（3 条 · 必须给）

跟值得投同样的维度反向看。**这部分是这个流程的灵魂**——市面工具都不愿意说，但用户最需要这个：

- **Level / 薪资**：是不是横跳或降级？
- **Gap 风险**：Match Score 里"难补"的 Gap 会不会成为 reject reason？
- **画像风险**：用户从 X 跳到 Y 招聘经理会怎么质疑？（参考 Match Score 里"风险"部分）
- **公司 / 团队风险**：阶段是不是不稳？团队是不是刚换 lead？业务是不是在转型？
- **行业 / 方向风险**：JD 描述的方向是不是过热 / 过冷 / 已经晚了？
- **入职后真实工作**：JD 写的跟实际可能差距大（如"design leadership"实际是"画原型"）
- **机会成本**：流程长 / 拿面概率低 / 用户手里有更好的

```
❌ 不值得投 1：Level 标的是 Senior 但 JD 职责描述更接近 Lead，可能错位
❌ 不值得投 2：JD 强调 ML/coding，用户简历这块只有 partial 覆盖，过 ATS 难
❌ 不值得投 3：公司 Glassdoor 6 个月内换过 Design Head，team 处于动荡期
```

如果真没有 3 条 → 老实说"我从 JD 看不到明显 downside，但建议查这些盲区：① 公司财务健康度 ② Glassdoor 离职率 ③ 团队成员 LinkedIn 任期"。

### ④ 预估拿面概率（给区间）

依据 [../frameworks/go-no-go.md](../frameworks/go-no-go.md) 的估算法。核心因子：

- Match Score（基础概率）
- 是否有内推（巨大乘数）
- 简历是否 tailor（中等乘数）
- 公司当前招聘热度（如果知道）
- 用户画像是否有 deal-breaker（直接砍半）

```
预估拿面概率：30-45%
（裸投 + 简历未 tailor 基线约 15-25%；
 你的 Match 78% 拉到 30-45%；
 如果能找到内推，可拉到 50-65%）
```

**永远给区间 + 给"如何提升"的具体动作。**

### ⑤ 建议下一步（具体动作，不要废话）

从这几个动作里挑最该做的 1-2 个：

- 🚀 直接投（裸投也能拿面）
- 🔧 先 Tailor 再投（去 Tailor）
- 🕵️ 先找内推（具体怎么找）
- 📚 先补 1-2 个关键 Gap（如重写某段经历 / 加某个 portfolio case）再投
- ⏸ 暂时不投（明确理由）
- 🔍 先做尽调（找前员工聊 / 看更多公开资料）

```
🎯 建议下一步：
1. 优先找内推（LinkedIn 搜该公司 Design 同事，提 referral）
2. 同步把跑一版 tailored 简历
3. 投递前再用面试预测看一下面试可能题，准备 Top 5
```

## Step 3 · 沉淀 + 生成 HTML 报告 + 导流

1. **更新 JD 文件**：加 should_i_apply section（推荐指数 + 决策 + 下一步动作）。
2. **生成 HTML Offer Strategy Report（默认动作）** —— 把 解码 + 匹配 + 面试预测 + Should-I-Apply + 策略层（内推 / portfolio / 时间线）融合成一份可分享的单文件 HTML：
   - 路径：`~/Desktop/Claude skills/offer-strategy-<company>-<role>-<YYYYMM>.html`（目录不存在也先建）
   - 完整 20 章骨架 / 视觉规范 / 内容铁律见 [../frameworks/offer-strategy-report.md](../frameworks/offer-strategy-report.md)
   - CSS / 结构骨架参考 [../examples/offer-strategy-template.html](../examples/offer-strategy-template.html)
   - **生成后告诉用户文件路径**，并在 JD 文件末尾加一行 `> 📊 HTML 报告：~/Desktop/Claude skills/offer-strategy-<slug>.html`
   - 如果匹配还没跑或者简历事实不足以填 6 条 Must Have 证据 → **拒绝生成**，告诉用户"先补完 ___ 再出报告"
3. **导流**：
   > "评估完成 ✅，报告在 `~/Desktop/Claude skills/offer-strategy-<slug>.html`。基于上面的建议：
   > - 决定投 → 跑一版 Tailor 简历
   > - 还想多看一份对比 → 把另一份 JD 发我
   > - 决定不投 → 我帮你把这份归档到 jd-bank（status: closed）"

## 常见卡点

| 症状 | 处理 |
|---|---|
| 用户已经决定要投 | 告诉用户："那直接走 Tailor。但建议先看一眼我这里的 ❌ 部分，避免盲区。" |
| 用户犹豫到底投不投 | 不要逼用户决定。给清楚信息后说："要不要先把这份存进 jd-bank（status: considering），过两天再决定？" |
| 用户说"你帮我决定" | **温和拒绝**："最终决策需要你自己的硬约束和直觉。我能做的是把信息摆清楚——基于我看到的，倾向是 ___（推荐指数对应方向）。" |
| 用户问"我能不能拿到 offer" | 老实说：「拿面概率我能估，拿 offer 受面试表现和 fit 影响太大，估了也不准。先把拿面概率最大化。」 |
